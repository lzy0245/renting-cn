`租房.skill` 是一个面向租房决策的 Codex Skill。它把小红书真实体验、房源平台信息、地图通勤、现场看房、签约权属和押金条款放到同一个判断框架里，帮你少踩坑、少被催、少为含糊条款买单。

说明：`租房.skill` 是这个项目面向中文用户的展示名，`renting-cn` 是 Codex 技能目录名和调用名。使用时可以说“使用 renting-cn 技能 ...”。

租房很少输在“没看到房源”。真正麻烦的是信息混在一起：中介说的、平台写的、小红书吐槽的、现场看到的、合同里藏的，可信度完全不同。`租房.skill` 的目标是把这些信号拆开，逐条核验，再给出下一步动作。

## 完整使用案例：杭州 EFC 实习租房

这个案例来自一次真实需求：要去杭州实习，预算很紧，网上信息又杂，总是在地图软件、租房平台、小红书之间来回切换。于是把这件折磨人的事交给 `租房.skill`：杭州 EFC 上班，预算 1300，不合租，押一付一，通勤 30 分钟内。

### 用户输入

```text
使用 renting-cn 技能：帮我找房子，我在杭州 EFC 上班，预算 1300，不要合租，押一付一，通勤 30 分钟内。
```

```text
|---|---:|---:|---|

不要优先找 EFC 楼下、欧美金融城公寓、奥克斯未来中心这类核心盘。

可直接发送的话术：
你好，我在 EFC 上班，预算 1300 以内，只接受不合租，最好独门独卫/独厨卫，押一付一，通勤 30 分钟内。

想先确认：
1. 这套具体在哪个小区/公寓？到 EFC 创景路站大概多久？
2. 是整租独立房间，还是隔断单间？卫生间和厨房是否独立？
3. 月租、押金、付款周期、水电网费、物业费、服务费分别是多少？
4. 签约方和收款方是谁？能否和房东本人签合同？
5. 押金什么情况下扣，退租后多久退？
6. 能否发今天拍的完整视频：楼栋入口、楼道、房间、卫生间、厨房/电磁炉、窗外、水电表？视频里请避开或打码门牌号、身份证件、合同、支付码和其他个人信息。

没确认签约方和收款方，不付。
没写清押金扣除和退还规则，不付。
没确认是否隔断、是否独立厨卫、是否民用水电，不付。
被催“先付定金留房”，不付。
```

### Skill 执行过程

1. 先定位通勤中心：杭州 EFC / 欧美金融城在余杭未来科技城、创景路站一带。
2. 判断预算现实性：1300、整租、押一付一、30 分钟到 EFC 是紧约束，EFC 核心区正规一室一厅大概率超预算。
3. 用小红书 MCP 做只读检索，优先查找近期租房线索、求租反馈和避雷信号。
4. 用公开网页补充位置、地铁和房源价格带。
5. 把小红书信号分成三类：可追房源、区域建议、风险提醒。
6. 输出区域优先级、候选线索、追问话术和付款前底线。

## 适合谁

- 正在北上广深杭成等城市租房，想先筛掉明显高风险房源的人。
- 需要结合小红书避雷、评论区反馈和房源信息做判断的人。
- 第一次租房，分不清房东、中介、二房东、转租、长租公寓风险的人。
- 看房前想要一份能直接照着问、照着拍、照着查的清单。
- 签合同前想检查押金、维修、提前退租、付款方、签约方和授权证明的人。

## 它能做什么

### 小红书租房研究

生成精确搜索词，判断笔记类型，提取评论区重复信号，识别中介推广、转租、室友招租、长租公寓宣传、避雷帖和真实居住体验。

如果环境里配置了只读 Redbook MCP，它可以使用：

- `search_notes`
- `get_note_content`
- `get_note_comments`

它不会自动评论、私信、引流、批量抓取，也不会把小红书当作权属、付款安全或合同有效性的证明。

### 房源筛选

把房源拆成几个核心问题：

- 总成本：租金、押金、付款周期、中介费、服务费、物业、水电燃气、网费、清洁费。
- 可信度：价格是否明显低于附近同类房源，低价原因是否合理。
- 渠道和权属：发帖人、签约方、收款方、产权或授权是否能对上。
- 居住条件：采光、通风、噪音、潮湿、虫害、设备、楼道、电梯、消防和小区环境。
- 合同风险：押金扣除、维修责任、提前退租、续租涨价、转租规则、交割清单。

### 看房准备

把平台和小红书里的模糊信号转成现场动作：

- “隔音差”变成开窗关窗噪音测试。
- “退押金难”变成押金扣除标准和退还时间追问。
- “物业差”变成电梯、消防、快递、门禁、维修响应检查。
- “中介避雷”变成费用表、签约方、授权证明和书面承诺核验。

### 合同和付款风险检查

签约前重点检查：

- 谁有权签。
- 钱打给谁。
- 合同地址、房号、房间、公共区域是否写清。
- 押金能因为什么扣，多久退。
- 维修谁负责，多久响应。
- 提前退租是否扣押金，是否必须找替租。
- 口头承诺是否进入合同或补充协议。

这不是法律意见。它提供风险提示、追问清单、条款提示和证据准备步骤。

## 决策框架

`租房.skill` 使用四条证据线：

| 证据线 | 主要用途 | 不适合证明 |
|---|---|---|
| 小红书 / Redbook | 居住体验、反复投诉、区域口碑、中介或公寓品牌风评 | 权属、付款安全、当前可租 |
| 房源和平台信息 | 租金、户型、照片、费用、可入住时间 | 隐藏缺陷、真实授权 |
| 地图和公开信息 | 通勤、周边、政策、市场背景 | 具体房间状态 |
| 线下和文件 | 房屋现状、签约方、付款方、押金条款、交割证据 | 社区长期体验 |

推荐倾向分为：

- 推荐看房
- 谨慎推进
- 先追问再看
- 暂不建议
- 信息不足

## 隐私和安全边界

使用前请打码：

- 姓名、手机号、身份证号。
- 门牌号、房号、合同编号。
- 支付账户、银行卡、收款码。
- 聊天截图里的头像、昵称和联系方式。

不要在以下情况付款：

- 没看过 exact 房源。
- 签约方、收款方、发帖人关系不清。
- 对方拒绝提供产权、授权或转租同意。
- 总费用、押金扣除、提前退租规则含糊。
- 被催促“先付定金留房”。

## 可选：接入小红书 MCP

如果你已经配置了 [Redbook-Search-Comment-MCP2.0](https://github.com/chenningling/Redbook-Search-Comment-MCP2.0)，`租房.skill` 会优先使用只读检索工具收集少量高相关样本。

推荐开启或暴露这些工具：

- `search_notes`
- `get_note_content`
- `get_note_comments`

不建议在租房研究场景里开启这些能力：

- 自动评论。
- 智能评论发布。
- 私信或引流。
- 批量抓取。
- 高频轮询。

推荐采样策略：

- 2 到 4 组关键词。
- 5 到 12 条相关结果。
- 优先看近 3 到 6 个月内容。
- 对强烈避雷、具体小区、中介、品牌和低价房源读取评论区。

没有 MCP 也可以用。它会给出搜索词、优先打开的内容、要复制回来的字段和非小红书核验路径。

## 小红书能力来源与改进

`租房.skill` 的小红书检索能力适配了 [chenningling/Redbook-Search-Comment-MCP2.0](https://github.com/chenningling/Redbook-Search-Comment-MCP2.0)。上游项目是一个基于 Playwright 的小红书 MCP Server，提供登录、关键词搜索、笔记内容获取、评论获取，以及 AI 评论生成/发布等能力。

`租房.skill` 对它的使用方式做了收敛：只把它作为小红书资料检索入口，不把它当成评论发布、私信引流或账号运营工具。租房场景里默认只使用这些只读能力：

- 搜索小红书笔记：`search_notes`
- 获取笔记正文：`get_note_content`
- 获取评论区信号：`get_note_comments`

在此基础上，`租房.skill` 做了租房研究和决策层改进：

- 把小红书搜索改成租房场景关键词矩阵，例如“城市 + 区域 + 租房 + 避雷”“小区 + 隔音 / 潮湿 / 虫 / 物业”“品牌公寓 + 押金 / 维修 / 退租”。
- 给笔记做角色分类，区分真实居住体验、避雷帖、中介推广、转租、室友招租、长租公寓宣传和不明来源内容。
- 给评论区信号分权重，优先看反复出现、可线下验证、和具体小区/中介/品牌有关的反馈。
- 明确禁止把小红书当成产权、授权、付款安全、当前可租或合同有效性的证明。
- 把平台信号转成下一步动作：看房要测什么、问中介什么、合同里要写什么、付款前必须核验什么。
- 默认避开自动评论、智能评论、私信引流和高频抓取能力，减少账号、隐私和平台规则风险。

换句话说，上游 MCP 解决“怎么把小红书内容取回来”，`租房.skill` 解决“取回来以后，如何用于租房判断”。

## 为什么值得收藏

租房信息很碎，很多坑只有住过的人知道，很多风险又必须靠线下和文件确认。`租房.skill` 不追求一句话替你拍板。它更像一个冷静的陪看人：先把信息分层，再把不确定性变成能问、能查、能拍、能写进合同的动作。

它尤其适合做三件事：

- 看房前，把明显不靠谱的房源挡掉。
- 签约前，把钱、权、责问清。
- 纠结时，把“感觉还行”改成有证据的比较。
